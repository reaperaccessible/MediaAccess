MediaAccess - Changelog

Version 2.50

New: A-B loop (audio). Set the loop start with Shift+[ and the end with Shift+], at the current playback position (the two keys right of P, regardless of keyboard layout). The L key toggles repeat on or off, and the key below Escape clears the loop. Page Up and Page Down jump playback to the loop start and end. Once both markers are set and the loop is enabled, playback returns to the start automatically when it reaches the end. The looped section repeats seamlessly and stays accurate even while you change speed or pitch during playback, regardless of the active tempo algorithm (SoundTouch, Speedy or Signalsmith). Looping is not available for video or live streams. Markers reset on every track change. Requested by Romain.

Version 1.92

Fix a single-instance race condition that split a multi-file Explorer selection across two MediaAccess windows. When you selected, say, 8 files of mixed extensions (.mkv, .mp4, .m4a, .mov, .mid, etc.) in Windows Explorer and pressed Enter, Explorer launched several MediaAccess.exe processes in parallel. The first to grab the single-instance mutex won, but its main window hadn't been created yet (BASS init, settings load, keymap parsing take a few hundred milliseconds). The other processes called FindWindowW too early, got nullptr, fell through, and each became its own autonomous instance — splitting your selection into two playlists in two separate windows. The latecomer launchers now poll FindWindowW for up to 3 seconds before giving up, which is more than enough time for the winner to finish CreateWindowExW. Result: a single multi-file selection now reliably lands in one playlist, in one window.

Version 1.91

Bundles four small accessibility improvements to the Options dialog, plus a documented limitation.

The tab control inside the Options dialog now exposes an explicit accessible name ("Category" / "Catégorie") to MSAA, so a screen reader announces "Category, tab" instead of an unnamed "tab". Initial keyboard focus is also explicitly placed on the tab control in WM_INITDIALOG instead of relying on whichever first WS_TABSTOP control Windows would have picked, which makes the dialog open with the active tab name spoken right away.

An empty LTEXT placeholder on the MIDI tab (the SoundFont status label, populated at runtime) now ships with default text in the resource template, so it can never be exposed to a screen reader as a static with no accessible name during the dialog's initial scan.

ShowTabControls is now called immediately after the tab items are inserted, before any EnableWindow or combo population call in WM_INITDIALOG. This means tab-specific controls are STATE_SYSTEM_INVISIBLE by the time any of them is enabled or disabled — screen readers ignore state-change events on invisible controls.

Finally, speechStop() is called at the very start of WM_INITDIALOG to drain any pending screen-reader announcement queued by the parent-window state change that DialogBox always triggers.

Known limitation: the residual "unavailable" / "non disponible" announcement that some screen readers (notably NVDA) emit when any modal dialog opens is inherent to the Win32 modal-dialog pattern — DialogBox disables the owner window, which fires an MSAA STATE_SYSTEM_UNAVAILABLE event on the main MediaAccess window, and assistive tech correctly verbalises it. Most Windows apps using DialogBox share this behavior. Eliminating it would require rewriting all our modal dialogs as modeless overlays, which is a much larger refactor; we have not done that yet. The four improvements above still make the rest of the dialog-open announcement noticeably shorter and clearer.

Version 1.85

Fix a regression reported by Sèb where opening a .ts video file from Windows Explorer silently did nothing. When the file was the second (or later) Explorer drop during the early startup window, the WM_COPYDATA handler pushed it into the playlist but never scheduled the batch-coalescing timer that triggers PlayTrack, so playback never started. The early-window detection now uses an explicit boolean flag flipped on in WM_CREATE and cleared by a one-shot timer after the coalescing delay, instead of GetTickCount() arithmetic against a startup timestamp — deterministic, no wraparound risk, and the batch path is only chosen when the timer is genuinely still pending.

Fix a companion to the v1.78 localized-fallback patch, also reported by Sèb. After pasting media from the clipboard or opening a file via Ctrl+O on the French build, the main window title could briefly show "Aucune lecture en cours" as the now-playing item until the first tag arrived. The v1.78 fix had been applied in PlayTrack but missed UpdateWindowTitle, which only compared the tag fallback against the English placeholders "No title" and "Nothing playing". The comparison now also covers the active language's localized versions, so the placeholder is never promoted into the title in any language.

Fix the bare "MediaAccess" window title that appeared after pressing Stop and then Play on a local file, reported by Sèb. Stop() used to clear the now-playing state unconditionally, but for a non-live local stream BASS keeps the channel alive (only the position is rewound) and Play() simply resumes that same channel without calling SetNowPlaying again — so the title ended up empty. Stop() now preserves the now-playing state for local non-live streams; it still clears it for live streams (radio, podcast streaming, YouTube live) and for video files since those paths free or hide the player and a subsequent Play would not have a meaningful title to keep. Stop→Skip and Stop→Open-new-file are unaffected because PlayTrack always installs a fresh now-playing entry on the next load.

Version 1.84

Fix the keymap merge logic that, since v1.75, prevented Ctrl+Shift+L (Cycle subtitles) from coming back on its own after the upgrade. v1.75 had moved that shortcut from Ctrl+Shift+T (which collided with Speak total) to Ctrl+Shift+L. For users whose personal keymap file still listed the old Ctrl+Shift+T, the v1.75 startup conflict resolver correctly dropped the duplicate but left an empty line behind for the action; the next "merge missing defaults" step mistook that empty line for a deliberate "unbind" by the user and refused to re-inject Ctrl+Shift+L. Version 1.76

Several improvements to video file playback, driven by user reports on TNT TV recordings and Captvty replay files.

The first bug only triggered when the "Load folder" option in Preferences was enabled. Picking a single video file through File > Open then left MediaAccess with an empty playlist and nothing playing, because the folder-expansion routine was filtering its scan to audio-only extensions — the video file the user had just picked was silently dropped on the way to the playlist. The workaround until now was to paste the file from Explorer with Ctrl+V, which bypasses the folder expansion. The expansion is now skipped when the picked file is not audio, so opening a video file works the same way regardless of whether "Load folder" is on or off.

The second bug affected .mp4 files that contain BOTH video and audio (typical Captvty TV replay downloads). MediaAccess deliberately routes .mp4 to its BASS audio engine because most .mp4 files in an audio library are M4B-style audiobooks where BASS gives you the full tempo/pitch/effects pipeline. But on a video .mp4, BASS would happily extract the audio track and the user would get sound only — the video stream silently lost. MediaAccess now probes the MP4 container's `moov` atom on load; if a video track is present, the file is routed to the libmpv video engine instead. Audio-only .mp4 files (and all .m4b audiobooks) keep the BASS audio path unchanged. A defensive secondary fallback also tries libmpv as a last resort if BASS rejects an .mp4 outright — covers rare cases where the probe missed (corrupted moov, unusual layout, etc.).

The libmpv video engine is now configured with more permissive ffmpeg options (analyzeduration=30s, probesize=100MB, fflags=+discardcorrupt+genpts, err_detect=ignore_err). These help with MPEG-TS captures that start mid-stream (live recordings) or contain non-standard private-data PIDs that confuse the demuxer's default analysis budget. Most TNT TV recordings now play correctly. A small subset of recordings with non-standard DVB private PIDs (codec types 0x0B / 0x0C) still cannot be played — those files would need to be pre-converted with an external tool. The cost of these new options on a healthy file is a few extra milliseconds of startup probing.

libmpv now also routes its own warnings and errors into the MediaAccess log file (%LOCALAPPDATA%\MediaAccess\mediaaccess.log). When a video file fails to play, the log captures the demuxer/codec error message — useful for future bug reports and diagnosis.

Version 1.52

Fix the auto-update silently failing to overwrite MediaAccess.exe because the running process held a file lock. Users on 1.49 / 1.50 / 1.51 saw the in-app updater download, run the installer, report success, then on the next launch find the same old version still installed and the same update offer pop up again. The installer now ships with CloseApplications=yes + AppMutex=MediaAccessSingleInstance so Inno Setup's Restart Manager closes the running MediaAccess gracefully before file copy. The updater also delays the installer launch by 2 seconds via a cmd wrapper to give the main window time to fully exit, so both ends of the chain are robust regardless of update-checker age. This is the lock-up bug, not a new feature — apologies to everyone who saw the loop.

Version 1.51

Fix the update checker offering an update over and over when MediaAccess was already up to date. The check previously also fired when versions matched but commit SHAs differed (a corner case for shipping same-version hotfix rebuilds), which surfaced false positives whenever an installer was built from a parent commit and uploaded under a release tag pointing at the next commit. The check now compares versions only — bumping the version is how a release ships, no surprises.
